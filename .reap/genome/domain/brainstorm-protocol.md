# Brainstorming Protocol

> Objective 단계에 통합된 설계 브레인스토밍 프로세스 규칙

## 개요

Objective Step 5에서 실행되는 구조화된 설계 프로세스.
"간단해서 설계 불필요"는 안티패턴 — 모든 목표가 brainstorming을 거친다 (간단한 건 짧은 설계).

## Brainstorming 단계 (Step 5a~5e + Step 8)

| 단계 | 내용 | 규칙 |
|------|------|------|
| 5a | 비주얼 컴패니언 제안 | 독립 메시지, evolve에서도 제안 (유저 거부 시만 스킵) |
| 5b | 명확화 질문 | **한 번에 하나**, 객관식 우선, 서술형은 최후 수단 |
| 5c | 접근법 탐색 | 2-3개 대안 + 트레이드오프 표 + 추천 |
| 5d | 섹션별 디자인 승인 | 아키텍처/컴포넌트/데이터흐름/에러처리/테스트 전략, 각 섹션 후 확인 |
| 5e | 스코프 분해 감지 | 독립 서브시스템 2+ → 분리 제안, FR 10개 초과 → 경고 |
| 8 | Spec 리뷰 루프 | subagent 검토, 최대 3회, 기준: 완전성/일관성/명확성/스코프/YAGNI |

## 비주얼 컴패니언

- **서버**: `server.cjs` — Node.js 내장 모듈 전용 (http, fs, crypto, url)
- **통신**: HTTP + WebSocket (RFC 6455), `.events` 파일 (JSON Lines)
- **프레임**: `frame.html` — content fragment 자동 래핑
- **타임아웃**: 30분 미사용 시 자동 종료
- **파일 위치**: `.reap/brainstorm/` (init/update 시 설치)

### 브라우저/터미널 판단

| 브라우저 | 터미널 |
|----------|--------|
| UI 목업, 와이어프레임 | 요구사항, 범위 질문 |
| 아키텍처 다이어그램 | 개념적 A/B/C 선택 |
| 나란히 비교 | 트레이드오프 표 |
| 디자인 폴리시 | 기술 결정 |

## Spec 리뷰 기준

| 카테고리 | 검토 항목 |
|----------|-----------|
| 완전성 | TODO, 플레이스홀더, 미완성 섹션 |
| 일관성 | 내부 모순, 상충하는 요구사항 |
| 명확성 | 잘못 구현할 여지가 있는 모호한 요구사항 |
| 스코프 | 단일 generation에 적합한 범위인가 |
| YAGNI | 요청하지 않은 기능, 과잉 설계 |
| 검증가능성 | 객관적으로 검증 불가능한 완료 기준 |

## Autonomous Override 동작

| 단계 | standalone | evolve |
|------|-----------|--------|
| 5a 비주얼 제안 | 제안 | 제안 (거부 시만 스킵) |
| 5b 질문 | 유저에게 질문 | 기존 컨텍스트로 자체 판단 |
| 5c 접근법 | 유저 선택 대기 | 추천 접근법 자동 선택 |
| 5d 디자인 | 섹션별 유저 승인 | 자동 진행 |
| 8 리뷰 | 자동 실행 | 자동 실행 |
