## 현재 동작
- `reap init` 시 genome 템플릿(placeholder)만 복사
- 첫 generation objective에서 genome이 placeholder면 사용자와 채워넣으라고만 안내
- preset 사용 시 미리 정의된 genome으로 초기화

## 개선 방향

### 1. Genome 개념 브리핑
- 셋업 시작 전 사용자에게 Genome이 무엇인지 설명:
  "Genome은 이 프로젝트의 설계와 지식을 기록하는 곳입니다.
   아키텍처 원칙, 비즈니스 규칙, 개발 컨벤션, 기술 제약 등
   프로젝트 내부에서 팀이 결정하고 통제하는 것들이 해당됩니다.
   세대를 거치며 진화하는 프로젝트의 DNA입니다."

### 2. Greenfield 프로젝트 (신규)
- 대화형으로 수집 (순서 중요):
  1. 먼저 앱의 소개를 묻기:
     - "어떤 애플리케이션을 만들고 싶으세요? 간단히 소개해주세요."
     - "주요 사용자는 누구인가요? 어떤 문제를 해결하나요?"
  2. 앱 소개를 바탕으로 적절한 기술 스택을 추천:
     - 앱 유형(웹, API, CLI, 모바일 등)에 맞는 스택 제안
     - 2~3개 선택지를 장단점과 함께 제시
     - 사용자가 이미 선호하는 스택이 있으면 그것을 존중
  3. 스택 결정 후 추가 질문:
     - "아키텍처 원칙이나 설계 결정사항이 있나요?"
     - "개발 컨벤션이 있나요? (코딩 스타일, 브랜치 전략, 테스트 정책 등)"
     - "기술적 제약이 있나요? (성능 요건, 호환성, 의존성 제한 등)"
- 수집한 내용을 principles.md, conventions.md, constraints.md에 구조화하여 저장

### 3. Adoption 프로젝트 (기존 소스 기반)

#### 3-1. REAP Root 검증
- init 경로가 프로젝트 루트로 적절한지 검증:
  - 프로젝트 루트 시그널 확인: package.json, go.mod, Cargo.toml, pom.xml, .git 등
  - 서브폴더에서 init한 경우 감지: 상위 디렉토리에 프로젝트 루트 시그널이 있으면 경고
    "현재 위치가 서브폴더인 것 같습니다. 상위의 /path/to/project-root 가 프로젝트 루트 아닌가요?"
  - 모노레포 내 특정 패키지에서 init한 경우 감지: 상위에 workspace 설정(pnpm-workspace.yaml, lerna.json 등)이 있으면 안내
    "모노레포 구조가 감지되었습니다. 전체 루트에서 init할지, 이 패키지에서만 init할지 선택해주세요."
  - 프론트엔드/백엔드 중 한쪽만 지정한 경우: 동일 레벨에 다른 서브프로젝트가 있으면 안내
    "같은 레벨에 /server, /shared 디렉토리가 있습니다. 상위 디렉토리에서 전체 프로젝트로 init하는 게 맞을 수 있습니다."
- 사용자 확인 후 진행 (사용자가 의도적으로 서브폴더에서 init하는 경우도 허용)

#### 3-2. 기존 문서 수집 및 검증
- 프로젝트 내 기존 문서를 탐색:
  - README.md, CONTRIBUTING.md, ARCHITECTURE.md, docs/ 등
  - CLAUDE.md, AGENTS.md 등 AI 에이전트용 문서
  - ADR(Architecture Decision Records) 디렉토리
- 발견된 문서를 사용자에게 보여주고 현행화 여부 확인:
  "다음 문서들을 발견했습니다. 각각 현재 상태를 반영하고 있는지 확인해주세요."
  - 문서별로 "최신/부분적으로 오래됨/더 이상 유효하지 않음" 확인
  - 최신인 문서 → genome 추론 소스로 활용
  - 오래된 문서 → 참고만 하되 코드 기반 분석을 우선

#### 3-3. 기존 소스 분석
- 코드베이스를 분석하여 genome을 자동 추론:
  - package.json, tsconfig, Dockerfile 등에서 기술 스택 파악
  - 디렉토리 구조에서 아키텍처 패턴 추론
  - lint/formatter 설정에서 컨벤션 추출
  - 테스트 설정에서 테스트 전략 파악
- 검증된 기존 문서 + 코드 분석 결과를 종합하여 genome 초안 작성
- 추론 결과를 사용자에게 보여주고 확인/수정 받기:
  "기존 문서와 소스를 분석한 결과입니다. 맞는지 확인해주세요."
- 사용자 확인 후 genome 파일에 반영
- 코드와 문서에서 파악할 수 없는 부분은 대화형으로 추가 질문

## 기대 결과
- 신규 프로젝트: 대화만으로 genome이 자연스럽게 채워짐
- 기존 프로젝트: 소스 분석 + 대화로 현실에 맞는 genome 구축
- placeholder 상태로 시작하는 경우가 없어짐
